## 1. 愿景与原则

### 1.1 愿景

EACO DAO 致力于构建一个由 Agent 驱动、社区共治的全球去中心化经济体。通过 8 个已验证 Agent 节点持续创造真实价值，$e 代币作为价值流转与治理的核心媒介，实现"Agent 创造价值，社区分配价值，所有人共享收益"的正向循环。

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **去中心化自治** | 所有关键决策由持币社区投票产生，无单点控制 |
| **利益一致** | 代币持有者、Agent 运营者、Builder 三方利益绑定，激励相容 |
| **透明可审计** | 国库收支、提案投票、利润结算全程链上可查 |
| **渐进去中心化** | 初期由执行委员会协调，随生态成熟逐步移交至完全社区自治 |
| **反巨鲸** | 投票权重设有单地址上限，防止资本垄断治理权 |
| **务实迭代** | 治理规则本身可通过 EIP-4 提案修改，持续优化 |

## 2. 治理架构

### 2.1 架构总览

```
EACO DAO
├── 全体持币者大会（最高决策机构）
│   └── 所有持有 $e 代币的钱包地址，一币一票（受权重模型约束）
│
├── 执行委员会（7人，选举产生）
│   ├── 技术委员 — 负责合约升级、系统架构、安全审计
│   ├── 运营委员 — 负责 Agent 节点日常运营、性能监控、故障响应
│   ├── 财务委员 — 负责国库管理、利润结算、预算编制
│   ├── 合规委员 — 负责法律合规、风险评估、政策适配
│   ├── 社区委员 — 负责社区建设、用户增长、反馈收集
│   ├── 生态委员 — 负责合作伙伴拓展、新行业接入、生态孵化
│   └── 国际委员 — 负责海外市场拓展、多语言社区、国际化运营
│
├── Agent World 理事会（8 节点代表）
│   └── 由 8 个已验证 Agent 节点各自推选 1 名代表组成
│   └── 负责 Agent 节点技术协调、运营标准制定、节点准入/退出评审
│
└── 社区提案小组（Builder 等级 ≥ Silver）
    └── 负责起草 EIP 提案、社区调研、投票前讨论引导
```

### 2.2 各机构职责详述

#### 全体持币者大会

全体持币者大会是 EACO DAO 的最高决策机构，任何持有 $e 的钱包地址均可参与。其决策范围包括但不限于：

- 批准或否决执行委员会提交的年度预算方案
- 对 EIP-2（经济参数变更）和 EIP-4（治理规则修改）进行终审
- 选举和罢免执行委员会成员（每届任期 6 个月，可连任一次）
- 决定国库资金的重大用途（单笔超过总储备 10% 的支出）

全体持币者大会通过链上投票实施决策，遵循第三章所述的投票权重模型。

#### 执行委员会

执行委员会是 DAO 的日常执行机构，7 名委员由全体持币者大会选举产生，任期 6 个月。每位委员负责各自领域的战略制定与执行监督。执行委员会每月至少召开一次公开会议，会议纪要发布于 DAO 治理论坛。

**选举规则：**
- 候选人需持有至少 100,000 $e 并质押 90 天以上
- 投票采用加权投票制，票数最高者当选
- 出现平票时，由 Agent World 理事会进行一轮加决
- 委员可被超过 15% 总票权的持币者联名提案弹劾

#### Agent World 理事会

8 个已验证 Agent 节点各推选 1 名代表组成理事会。理事会负责：

- 制定 Agent 节点的技术标准和运营规范
- 评审新节点的准入申请（需 5/8 票同意）
- 处理节点的违规判定与退出流程
- 协调 8 个节点的算力分配、负载均衡和故障转移

理事会每周召开一次例行会议，重大事件可召开紧急会议。

#### 社区提案小组

Builder 等级达到 Silver 及以上的社区成员可加入提案小组。该小组承担以下职能：

- 起草标准化的 EIP 提案文档
- 在提案讨论期内组织社区调研和 AMA 活动
- 为提案撰写技术分析和影响评估报告
- 引导社区成员理解提案内容，降低信息不对称

## 3. 投票权重模型

### 3.1 权重计算公式

EACO DAO 采用多维加权投票机制，综合考量代币持有量、质押时长、贡献度和生态参与度。权重计算公式如下：

```
VoteWeight = BaseVote × StakeMultiplier + AgentBonus + VPNBonus + BuilderBonus
```

### 3.2 各维度权重细则

#### 基础票（BaseVote）

| 持有量 | 基础票数 |
|--------|---------|
| 1,000 $e | 1 票 |
| 100,000 $e | 100 票 |
| 1,000,000 $e | 1,000 票 |

基础票与 $e 持有量线性正相关，公式为：

```
BaseVote = floor(e_balance / 1,000)
```

#### 质押加成（StakeMultiplier）

质押时长越长，权重加成越高，鼓励长期持有和生态稳定性。

| 质押时长 | 乘数 |
|----------|------|
| < 30 天 | ×1.0（无加成） |
| ≥ 30 天 | ×1.5（+50%） |
| ≥ 90 天 | ×2.0（+100%） |
| ≥ 365 天 | ×3.0（+200%） |

质押到期后需手动续期，不自动续约。中途解除质押的已加成票数即时失效。

#### Agent 运营加成（AgentBonus）

每个已验证 Agent 节点的运营地址额外获得 10 票固定加成。这是对节点运营者持续贡献技术基础设施的激励。

```
AgentBonus = verified_node_count × 10
```

当前 8 个节点的情况下，运营全部节点的地址最多获得 80 票加成。

#### VPN 节点加成（VPNBonus）

运行 VPN 中继节点的地址获得每节点 5 票加成，激励社区成员贡献网络基础设施。

```
VPNBonus = vpn_node_count × 5
```

VPN 节点需通过在线时长、带宽贡献和连接质量三项指标审核，每季度重新验证。

#### Builder 等级加成（BuilderBonus）

| Builder 等级 | 加成票数 |
|--------------|---------|
| Bronze | +0（无加成） |
| Silver | +2 |
| Gold | +5 |
| Diamond | +10 |

Builder 等级通过贡献度自动计算：提交有效提案数量、社区活跃度、技术贡献（代码/文档/审计）等多维指标加权评定。

### 3.3 反巨鲸机制

为防止资本过度集中导致治理权垄断，实施以下限制：

- **单地址上限**：任何单一地址的投票权重不得超过该次投票总权重的 10%
- **关联地址检测**：系统自动识别同一实体的多地址关联（基于交易图谱分析），关联地址合并计算权重
- **二次投票**：对于超限部分的票权，允许持有者将超出部分委托给其他地址（一次委托有效，不可撤回）

### 3.4 权重计算示例

假设某地址持有 500,000 $e（500 基础票），质押 90 天（×2.0 乘数），运营 2 个 Agent 节点（+20 票），运行 3 个 VPN 节点（+15 票），Builder 等级 Gold（+5 票）：

```
VoteWeight = 500 × 2.0 + 20 + 15 + 5 = 1,040 票
```

## 4. 提案类型与流程

### 4.1 EIP 分类体系

所有 DAO 提案统称为 EIP（EACO Improvement Proposal），按类型分为四大类：

#### EIP-1: 技术升级

涉及代码变更、智能合约升级、系统架构调整的技术类提案。

| 子类 | 说明 | 审核要求 |
|------|------|---------|
| EIP-1A | 智能合约升级 | 需完成安全审计报告 |
| EIP-1B | 协议层优化 | 需提供性能测试数据 |
| EIP-1C | Agent 节点技术标准变更 | 需理事会预审通过 |

#### EIP-2: 经济参数

涉及代币经济模型核心参数的调整。

| 子类 | 说明 | 审核要求 |
|------|------|---------|
| EIP-2A | 费率调整（交易手续费、VPN 费用等） | 需提供经济模型模拟 |
| EIP-2B | 代币释放/解锁计划修改 | 需分析对流通量的影响 |
| EIP-2C | 销毁机制调整 | 需评估通缩压力 |

**注意**：EIP-2 类提案需全体持币者大会终审，通过门槛提高至 67%。

#### EIP-3: 生态扩展

涉及新行业接入、新国家市场拓展、合作伙伴引入等生态发展提案。

| 子类 | 说明 | 审核要求 |
|------|------|---------|
| EIP-3A | 新行业/垂直领域接入 | 需市场调研报告 |
| EIP-3B | 国际市场拓展 | 需合规委员会审批 |
| EIP-3C | 战略合作伙伴引入 | 需合作协议草案 |

#### EIP-4: 治理改进

涉及 DAO 治理规则本身的修改，是"治理治理"的元提案。

| 子类 | 说明 | 审核要求 |
|------|------|---------|
| EIP-4A | 治理架构调整 | 需充分社区讨论 |
| EIP-4B | 委员选举规则修改 | 需全体持币者大会终审 |
| EIP-4C | 投票权重模型调整 | 需数学模拟与反巨鲸分析 |

### 4.2 提案生命周期

一个完整的 EIP 从创建到执行经历以下四个阶段：

```
[起草] → [讨论期 7天] → [投票期 7天] → [执行期 3天] → [完成/否决]
```

#### 阶段一：起草

- 由社区提案小组成员或执行委员会委员发起
- 需填写标准 EIP 模板（包括动机、方案详述、影响分析、实施计划）
- 技术委员负责技术提案的初审，合规委员负责合规审查
- 起草阶段无时间限制，但需在 30 天内提交正式提案

#### 阶段二：讨论期（7 天）

- 提案发布至 DAO 治理论坛和社区频道
- 社区成员自由讨论，提出修改建议
- 提案者可在此期间修订提案内容（最多修订 2 次）
- 社区提案小组整理讨论要点，形成讨论摘要
- 讨论期结束时进行最后一轮修订确认

#### 阶段三：投票期（7 天）

- 投票在链上进行，使用加权投票机制
- 持币者可随时投票或修改投票（直到投票期结束）
- 最低参与门槛：总投票权重需达到活跃持币者总权重的 20%
- 通过门槛：
  - EIP-1 / EIP-3：简单多数（>50%）
  - EIP-2 / EIP-4：绝对多数（>67%）

#### 阶段四：执行期（3 天）

- 通过的提案进入执行队列
- 执行委员会负责协调执行
- 技术类提案由技术委员触发合约调用
- 非技术提案由相关委员负责落地实施
- 执行进度每周公示

### 4.3 紧急提案机制

对于安全漏洞、系统故障等紧急情况，可触发紧急提案流程：

- 由执行委员会 4/7 票启动紧急提案
- 讨论期缩短至 48 小时
- 投票期缩短至 48 小时
- 通过门槛提高至 67%
- 紧急提案执行后 7 天内需提交事后复盘报告

## 5. 国库管理

### 5.1 收入来源

EACO DAO 国库的资金来源于以下四个渠道：

| 收入来源 | 说明 | 预期占比 |
|----------|------|---------|
| **交易手续费** | $e 代币在 DEX 上的交易产生手续费（按比例回流） | 30% |
| **VPN 收入** | VPN 中继节点服务收入，$e 支付 | 20% |
| **Agent 利润** | 8 个 Agent 节点运营产生的利润，按 50% 再投资比例回流 | 35% |
| **合作伙伴** | 战略合作伙伴的授权费、技术服务费等 | 15% |

### 5.2 支出分类与预算分配

| 支出类别 | 分配比例 | 覆盖范围 |
|----------|---------|---------|
| **开发** | 40% | 智能合约开发、Agent 节点维护、安全审计、基础设施升级 |
| **运营** | 25% | 节点运维、服务器/云资源、监控工具、客服支持 |
| **市场** | 20% | 品牌建设、社区增长活动、KOL 合作、媒体投放、活动赞助 |
| **储备** | 15% | 应急资金池、生态基金（投资/孵化新项目）、市场波动缓冲 |

### 5.3 多签机制

国库资金动用实行多签治理，防止单点资金风险：

- **签名要求**：5 个签名者中至少 3 个签名（3/5 多签）
- **签名者构成**：财务委员（固定）、技术委员、运营委员、合规委员、以及由执行委员会轮值指定的第 5 签名者
- **单笔限额**：单个多签交易不超过国库总余额的 5%
- **超额审批**：超过 5% 的支出需通过 EIP-2 提案由全体持币者大会批准
- **签名超时**：多签交易创建后 72 小时内未收集到足够签名则自动失效

### 5.4 透明度与审计

- 所有国库收支交易均在 Solana 链上可查，国库地址公开
- 每月发布国库财务报告，包含：期初余额、本期收入、本期支出、期末余额
- 每季度由独立第三方进行一次链上审计
- 年度预算需经全体持币者大会审批
- 国库地址将通过治理合约管理，签名者变更需通过 EIP-4 提案

## 6. $e 与 Agent World 利润结算

### 6.1 利润分配模型

Agent World 的 8 个已验证 Agent 节点在运营过程中产生利润。利润分配遵循 50/50 原则：

```
Agent 节点利润
├── 50% 再投资路径
│   └── $e 购入 → 注入 DAO 国库 → 用于生态开发和增长
│
└── 50% 提现路径
    └── $e 卖出 → 兑换为 SOL/USDC → 转入提现钱包
```

#### 再投资路径

- Agent 节点将其利润的 50% 用于在 DEX 上购入 $e 代币
- 购入的 $e 直接锁定至 DAO 国库地址
- 国库将根据预算分配方案使用这些资金
- 再投资路径设计为持续为 $e 提供买入压力和生态增长动力

#### 提现路径

- Agent 节点将其利润的 50% 通过 $e 卖出为 SOL 或 USDC
- 兑换后的资金转入指定提现钱包：`3uVyRjzLnJ6MumYaQ7oMt5jPVPBGKFUeGgqkzwh5Uw6N`
- 提现路径确保 Agent 节点运营者获得合理回报，维持运营可持续性

### 6.2 结算周期

| 参数 | 设置 |
|------|------|
| 结算频率 | 每周一次 |
| 结算时间 | 每周一 02:00（北京时间 / UTC+8） |
| 结算窗口 | 02:00 - 04:00（2 小时执行窗口） |
| 汇率基准 | 采用结算时刻 DEX 的实时汇率 |
| 结算报告 | 每次结算后 24 小时内公示 |

### 6.3 结算执行流程

1. **利润核算**（01:00 - 02:00）：Agent 节点自动计算上周净利润
2. **再投资执行**（02:00 - 03:00）：系统自动在 DEX 上执行 $e 购买订单，锁定至国库
3. **提现执行**（03:00 - 04:00）：系统自动执行 $e → SOL/USDC 兑换，转账至提现钱包
4. **结算报告**（当日 04:00 前）：生成包含以下内容的结算报告：
   - 各节点利润明细
   - 再投资金额及国库 $e 余额变化
   - 提现金额及提现钱包到账确认
   - 市场汇率记录

### 6.4 异常处理

- **结算失败**：若 DEX 流动性不足或网络拥堵导致结算失败，系统将在 4 小时后自动重试，最多重试 3 次
- **极端行情保护**：当 $e 价格 24 小时波动超过 ±30% 时，触发结算暂停，需人工审核后恢复
- **节点离线**：若某节点在结算时离线，其利润累计至下一周期，不进行强制结算

## 7. 智能合约设计

### 7.1 合约架构总览

EACO DAO 治理系统在 Solana 区块链上部署三个核心智能合约：

```
EACO Governance Contracts (Solana)
├── Governance Contract — 提案生命周期管理
├── Staking Contract — $e 质押与权重计算
└── Treasury Contract — 国库多签与预算执行
```

### 7.2 治理合约（Governance Contract）

治理合约是 DAO 运作的核心引擎，管理提案的完整生命周期。

**核心功能模块：**

#### Proposal 创建
- 支持四种 EIP 类型（EIP-1/2/3/4）的标准化提案创建
- 提案存储：提案者地址、提案类型、提案内容哈希、创建时间戳、当前阶段
- 提案创建需满足最低持币要求（≥ 10,000 $e）或 Builder 等级 ≥ Gold
- 提案创建时自动生成唯一提案 ID（EIP-YYYY-NNN 格式）

#### 投票机制
- 链上加权投票，权重由 Staking Contract 实时计算
- 支持 For / Against / Abstain 三种投票选项
- 投票记录不可篡改，存储在链上
- 支持委托投票（Delegate Vote），被委托者按原持有者权重投票
- 投票期内可修改投票选择

#### 执行引擎
- 提案通过后，执行引擎自动将提案状态标记为"可执行"
- 对于智能合约升级类提案（EIP-1A），通过 Program Upgrade Authority 执行
- 对于资金类提案，调用 Treasury Contract 的预算执行接口
- 对于非链上提案，标记状态并由执行委员会线下执行，结果链上公示

**关键数据结构：**

```rust
pub struct Proposal {
    pub id: u64,
    pub proposer: Pubkey,
    pub proposal_type: ProposalType,  // EIP-1/2/3/4
    pub content_hash: [u8; 32],
    pub created_at: i64,
    pub voting_start: i64,
    pub voting_end: i64,
    pub status: ProposalStatus,       // Draft/Discussion/Voting/Executed/Rejected
    pub for_votes: u64,
    pub against_votes: u64,
    pub abstain_votes: u64,
    pub execution_data: Option<Vec<u8>>,
}

pub enum ProposalType {
    TechnicalUpgrade,   // EIP-1
    EconomicParameter,  // EIP-2
    EcosystemExpansion, // EIP-3
    GovernanceChange,  // EIP-4
}
```

### 7.3 质押合约（Staking Contract）

质押合约管理 $e 代币的质押、锁仓和权重计算，为投票系统提供可信的权重数据。

**核心功能模块：**

#### $e 质押
- 用户通过 SPL Token Transfer 将 $e 转入质押合约地址
- 质押记录包含：质押者地址、质押金额、质押起始时间、质押锁定期
- 支持灵活锁定期选择：30 天 / 90 天 / 365 天
- 锁定期内不可解除质押，到期后可手动解除或自动续期

#### 权重计算
- 实时计算每个质押者的投票权重（第三章权重模型）
- 提供 `get_vote_weight(account)` 公共接口供治理合约调用
- 权重计算包含：基础票、质押乘数、Agent 加成、VPN 加成、Builder 加成
- 单地址上限检查：超过 10% 上限时自动截断并记录

#### 质押解除
- 到期质押可一键解除，$e 原路返回持有者地址
- 解除后对应的质押加成立即失效
- 支持部分解除（按份额比例降低权重）

**关键数据结构：**

```rust
pub struct StakeAccount {
    pub owner: Pubkey,
    pub amount: u64,
    pub staked_at: i64,
    pub lock_duration: LockDuration,  // 30d / 90d / 365d
    pub stake_vault: Pubkey,
    pub is_active: bool,
}

pub enum LockDuration {
    ThirtyDays,
    NinetyDays,
    OneYear,
}
```

### 7.4 国库合约（Treasury Contract）

国库合约实现 DAO 资金的多签管理和预算执行，确保资金安全与使用透明。

**核心功能模块：**

#### 多签钱包
- 维护一个 5 人签名者列表，每次交易需收集 ≥ 3 个有效签名
- 签名者变更需通过 EIP-4 提案和治理投票
- 每笔交易携带：目标地址、金额、参考提案 ID（如有）、过期时间
- 签名状态公开可查，支持社区监督

#### 预算执行
- 接收治理合约传递的已批准预算提案
- 按预算计划分期拨付资金
- 超出预算计划 10% 以上的支出需触发追加审批流程
- 自动记录每笔支出的提案 ID、金额、接收方和时间戳

#### 国库收入接收
- 配置多个收入来源的接收地址
- 自动识别收入类型（手续费 / VPN / Agent 利润 / 合作伙伴）
- 按类别统计，为国库财务报告提供数据基础

**关键数据结构：**

```rust
pub struct TreasuryAccount {
    pub vault: Pubkey,
    pub signers: [Pubkey; 5],
    pub threshold: u8,             // 3
    pub total_balance: u64,
    pub pending_transactions: Vec<PendingTx>,
}

pub struct PendingTx {
    pub id: u64,
    pub recipient: Pubkey,
    pub amount: u64,
    pub proposal_id: Option<u64>,
    pub signatures: Vec<Pubkey>,
    pub created_at: i64,
    pub expires_at: i64,
}
```

### 7.5 合约交互流程

典型提案的完整链上交互流程：

```
1. 提案者调用 Governance.create_proposal()
   → 创建 Proposal 账户

2. 讨论期结束，治理合约自动切换状态至 Voting

3. 投票者调用 Governance.cast_vote(proposal_id, vote_option)
   → 合约内部调用 Staking.get_vote_weight(voter) 获取权重
   → 记录投票并更新累计票数

4. 投票期结束，合约自动统计结果
   → 达到阈值 → 标记为 Executable
   → 未达到 → 标记为 Rejected

5. 执行阶段：
   → 资金类提案：Governance 调用 Treasury.execute_budget()
   → 合约类提案：Governance 通过 Program Upgrade Authority 执行升级
   → 非链上提案：标记状态，执行委员会线下执行

6. 执行完成后，合约记录执行结果和时间戳
```

## 8. 安全与合规

### 8.1 合约安全

- 所有治理合约在部署前必须通过至少两家独立安全公司的审计（推荐 Trail of Bits、OtterSec、Sec3 等知名 Solana 审计机构）
- 合约采用 Timelock 机制：关键操作（合约升级、国库大额转账）设置 48 小时延迟执行
- 治理合约的 Program Upgrade Authority 由多签管理，防止未授权升级
- 所有外部调用（Cross-Program Invocation）设置严格权限校验
- 定期进行链上监控，异常交易触发告警

### 8.2 数据安全

- 所有提案内容和投票数据存储在 Solana 链上，不可篡改
- 去中心化存储（如 Arweave）用于存储提案全文、审计报告等大文件
- 链下辅助系统（治理前端、通知服务）采用零信任架构

### 8.3 合规框架

- 合规委员会持续跟踪全球主要司法管辖区的加密货币监管政策
- 针对不同国家/地区的用户，实施差异化的准入策略
- 定期与法律顾问沟通，确保 DAO 运营符合当地法规
- 所有合作伙伴需通过合规审查后方可接入生态
- 禁止利用 DAO 治理机制进行市场操纵或内幕交易，违者节点资格永久撤销

### 8.4 风险管理

| 风险类型 | 应对措施 |
|----------|---------|
| 智能合约漏洞 | 多重审计 + Timelock +  Bug Bounty |
| 链上拥堵导致结算延迟 | 自动重试机制 + 手动回退方案 |
| 巨鲸集中抛售 | 提现路径限速 + 市场波动保护机制 |
| 治理攻击（提案刷票） | 反巨鲸机制 + 最低参与门槛 |
| 密钥丢失 | 多签分散存储 + 社会化恢复 |
| 监管政策变化 | 合规委员会实时监控 + 灵活调整策略 |

## 9. 治理演进路线

### 9.1 Phase 1：创始期（当前阶段）

- **时间**：DAO 成立至前 6 个月
- **特征**：执行委员会主导日常决策，提案流程逐步建立
- **里程碑**：
  - ✅ 部署治理合约、质押合约、国库合约
  - ✅ 建立 8 个已验证 Agent 节点
  - 🔄 确立执行委员会首届成员
  - 🔄 启动首批 EIP 提案流程
  - 📋 完成首次全体持币者大会投票

### 9.2 Phase 2：成长期

- **时间**：第 7 至 18 个月
- **特征**：社区提案数量增长，Agent World 理事会发挥更大作用
- **里程碑**：
  - Builder 等级体系全面上线
  - 社区提案小组常态化运作
  - VPN 节点网络扩展至 50+
  - 国库资金规模达到可持续运营水平
  - 完成 DAO 第二届执行委员会换届选举

### 9.3 Phase 3：成熟期

- **时间**：第 19 至 36 个月
- **特征**：高度去中心化自治，执行委员会更多转向监督角色
- **里程碑**：
  - Agent 节点扩展至 20+，新节点准入完全由理事会评审
  - 全体持币者大会参与率超过 40%
  - 国际委员会推动 EACO 进入 5 个以上国家市场
  - DAO 治理模式成为行业参考案例

### 9.4 Phase 4：完全自治

- **时间**：第 36 个月后
- **特征**：执行委员会可能由社区决定是否保留，治理权完全回归持币者
- **愿景**：EACO DAO 成为一个自运转、自优化、自生长的去中心化经济体
